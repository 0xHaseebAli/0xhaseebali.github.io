Investigation of Suspicious Login Activity Using SQL  
Project description:  
My organization is working to make their system more secure. I as a Junior Security Analyst  
tasked with auditing the internal database of an organization. My objective was to identify  
suspicious login activity and organize employee data to ensure security updates were applied  
correctly across departments.  
Retrieve after hours failed login attempts  
Identify failed login attempts occurring outside business hours to detect potential unauthorized  
access or brute-force activity.  
Query:  
SELECT * FROM log_in_attempts WHERE login_time > '18:00' AND success =  
FALSE;  
Query Breakdown:  
SELECT *, Retrieve all columns(event_id,username,login_date,login_time,etc).FROM  
log_in_attempts, Target the login activity table. login_time > '18:00', Filter attempts after  
business hours. success = FALSE, Show only Failed attempts.  
Retrieve login attempts on specific dates  
The query returned 19 failed login attempts recorded after 18:00. The earliest attempt was  
logged on 2022-05-08 at 22:38:31. To investigate further, I filtered for all login activity recorded  
on 2022-05-08 and 2022-05-09.  
Query:  
SELECT * FROM log_in_attempts WHERE login_date = '2022-05-09' OR login_date  
= '2022-05-08';  
Query Breakdown:  
SELECT, Retrieve all columns. FROM log_in_attempts, Point the login activity table. login_date  
= '2022-05-09', Filter attempts on '2022-05-09'. OR login_date = '2022-05-08', Uses the OR  
operator to include results from 2022-05-08.  
Retrieve login attempts outside of Mexico  
Query returned all 75 login attempts recorded across both dates, allowing the team to scope the  
investigation to the specific time frame. After investigating I believe there is an issue with the  
login attempts that occurred outside of Mexico. To identify potential unauthorized activity, I  
filtered for all login attempts originating from outside the organization's primary operating region  
(Mexico).  
Query:  
SELECT * FROM log_in_attempts WHERE NOT country LIKE 'MEX%';  
Query Breakdown:  
SELECT *, Retrieves all columns to provide full context (IP, User ID, Timestamp) for  
non-domestic logins. FROM log_in_attempts, Targets the primary login activity database.  
WHERE NOT, Exclude matching records. WHERE NOT country LIKE 'MEX%', Match both MEX  
and MEXICO entries  
Retrieve employees in Marketing  
Identify all Marketing department employees located in East office buildings to ensure their  
devices receive the required security updates.  
Query:  
SELECT * FROM employees WHERE department = 'Marketing' AND office LIKE  
'East%';  
Query Breakdown:  
SELECT *, Retrieve all columns.(employee_id, device_id, etc). FROM employees, Target  
employees table. WHERE department = 'Marketing', Filter Marketing staff only. AND office LIKE  
'East%', Match all East office locations eg 'East-170' and 'East-216'.  
Retrieve employees in Finance or Sales  
Identify all employees in the Finance and Sales departments to ensure their devices receive the  
required security updates.  
Query:  
SELECT * FROM employees WHERE department = 'Finance' OR department =  
'Sales';  
Query Breakdown:  
SELECT *, Retrieve all columns.(employee_id, device_id, etc). FROM employees, Target  
employees table. WHERE department = 'Finance', Filter Finance staff. OR department = 'Sales':  
Include Sales staff  
Retrieve all employees not in IT  
All non-IT departments required the same security updates. To achieve this I filter all  
departments excluding IT department.  
Query:  
SELECT * FROM employees WHERE NOT department = 'Information Technology';  
Query Breakdown:  
SELECT *, Retrieve all columns. FROM employees, Target the employee table. WHERE NOT,  
Exclude matching records. Department = ‘Information Technology”, Filter out IT staff.  
Notable:  
Several employees returned NULL device IDs, indicating unregistered devices that require  
immediate asset inventory follow-up.  
Summary:  
In this project, I queried two tables log_in_attempts and employees using AND, OR, NOT,  
LIKE, and wildcard % operators to support an internal security investigation. I filtered the  
log_in_attempts table to identify suspicious activity, including 19 after-hours failed logins  
and 144 access attempts from outside the organization's primary region. I also queried the  
employees table to scope targeted device update lists by department, enabling the IT team to  
prioritize and deploy security patches efficiently.
